# 当前项目重构结果总结 + 后续路线图

## 一、当前状态

项目核心定位：

- 本地 AI Web 工作台
- 本地模型代理层
- QQ 机器人接入
- 定时任务
- 工具调用与少量自动化能力

当前项目已经从“单个超大入口文件”开始向模块化过渡。

---

## 二、这轮重构已经完成的内容

### 已拆出的模块

- `server/server-cleanup.js`
  - 启动时清理日志、临时文件、浏览器 profile、失效 pid

- `server/server-bootstrap.js`
  - 数据初始化
  - 服务启动流程

- `server/server-connection-config.js`
  - 共享连接配置读取 / 保存 / 路由处理

- `server/server-http.js`
  - 静态资源服务
  - `/api/*` 代理转发

### 已完成的工程清理

- 删除大量运行日志与临时文件
- 清理 `data/*.tmp`
- 清理 `data/temp/`
- 清理浏览器 profile 缓存目录
- 修复 `README.md` 乱码，重写为可维护版本
- 清理 `server.js` 中 skills 上传 / 下载函数里 `return` 之后的死代码

### 已做的验证

- `node --check server.js`
- `node --check server/server-*.js`
- `npm test`
- 手动启动服务验证通过

当前结论：

> 本轮重构后，未发现明显功能回归。

---

## 三、当前仍然存在的问题

### 1. `server.js` 仍然偏大

虽然已经拆出基础设施层，但主文件依然承担了很多“业务胶水”职责。

### 2. `public/app.js` 体积很大

前端逻辑几乎集中在一个文件里，后续维护成本较高。

### 3. `server-qq.js` 过大

QQ 模块目前仍然是一个体量很大的核心文件，后续是重点拆分对象。

### 4. skills 相关历史代码仍有残留

当前状态不是“完全删除”，而是“部分停用 + 部分保留”：

- Web 端技能安装/管理接口大多已停用
- `skill-runner.ps1` 仍保留
- `skills/` 目录仍存在

这会让新维护者对功能边界产生误判。

### 5. 项目中仍可能有少量历史乱码文案

这次已经修复 README，但代码里可能还残留部分旧编码文本，尤其是历史 skills/旧提示文案。

---

## 四、后续维护建议

### 优先级 A：只做低风险高收益整理

适合“先稳定使用”的路线。

建议顺序：

1. 继续清理明显废弃的 skills Web 接口代码
2. 增加日志轮转 / 大小上限控制
3. 清点并修复剩余乱码文案
4. 补充 README 和维护文档

### 优先级 B：按功能开发路径拆模块

适合“后面还要继续开发”的路线。

原则：

> 不再为了拆而拆，而是在改某个功能前，先拆那个功能相关的代码。

例如：

- 要改聊天 → 先拆 chat/model glue
- 要改 QQ → 先拆 `server-qq.js`
- 要改定时任务 → 先拆 scheduler 周边 glue

---

## 五、推荐路线图

### 阶段 1：稳定维护期

目标：保证可用、降低脏数据和理解成本。

- [ ] 清理剩余废弃 skills 路由 / helper
- [ ] 加日志轮转
- [ ] 给 `logs/` / `data/` 增加更明确的目录说明
- [ ] 梳理是否需要保留 `skills/` 目录

### 阶段 2：结构优化期

目标：继续降低主文件复杂度。

- [ ] 抽离聊天 / 模型调用胶水逻辑
- [ ] 抽离通用响应 / 序列化辅助函数
- [ ] 逐步收缩 `server.js`

### 阶段 3：重点模块治理

目标：处理真正复杂的大块模块。

- [ ] 拆 `server-qq.js`
- [ ] 评估 `public/app.js` 前端模块化
- [ ] 明确 skills 系统最终保留还是彻底下线

---

## 六、对以后维护者的建议

### 改动前

- 先备份关键文件到 `.history/`
- 先跑：

```powershell
npm test
```

### 改动后

- 运行：

```powershell
node --check server.js
npm test
```

- 再手动验证：
  - 页面能打开
  - 聊天能发消息
  - 模型列表能读取
  - 定时任务能执行
  - QQ 配置页能打开

---

## 七、当前结论

这个项目已经从“能跑但偏乱”的状态，进入了“可以逐步维护”的状态。

当前最合理的策略是：

> 先稳住、先文档化、先清理废弃物；后续按功能需求继续拆，而不是一次性大重构。
